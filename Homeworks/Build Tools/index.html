Building JavaScript apps It sounds cool, but why do we need them?

Working on application development, there are repetitive tasks that take a long time to do them manually. We must automate these tasks, to be able to focus on the creation of our project. There are several tools that perform these tasks for us.

An advantage of using these tools is that it improves the performance of our application, our code is cleaner, the result code occupies less space, and at the same time help verify that we comply with the best coding practices. Also we save working time, by not having to deal with those tasks.

Module bundlers

Most projects of any scale will have their code split between a number of files. And while you could just include each file with an individual <script> tag, you should use a tool to automatically bundle everything into a single file. 

$ npm install -g browserify
$ browserify src/dev.js -o dist/script.js
browserify.org

Browserify is a tool for bundling Node packages for the browser.

This Node-centric philosophy has a number of benefits; bundling an app with Browserify is simple, you can using built-in Node modules like path, and you can use any code you’ve written for Node as is. The downside, of course, is that your Single Page App will generally need types of resources that Node apps won’t: CSS, images and fonts.

There are some plugins to handle all this, but using them to do something it wasn’t designed for the configuration file can get very messy and hard to maintain.

While Browserify is a wonderful and easy to use tool for bundling node modules for the browser, if you’re writing a Single Page App then you may prefer a tool designed for bundling anything, not just JavaScript code. 

$ npm install -g webpack
$ webpack src/dev.js dist/script.js
webpack.js.org

Webpack is a tool which takes JavaScript modules with dependencies and bundles them into files. It doesn’t care what dependencies they are, as long as they are JavaScript modules.

But for Webpack this means thinking about other files like CSS and Images as resource files.

Also with Webpack we can chain resources, which lets you do things like transforming your SCSS files to CSS first, and then transforming your CSS into a JavaScript module. It's magic!

Personally, the biggest downside of Webpack is that you are going to get enormous configuration files to do more than the basic tasks, and that is not very appealing to developers.

Task runners

Task runners are tools for defining and running tasks.
This means that anything you might manually do on the command line, it is done automatically. 

$ npm install -g grunt-cli
$ grunt
gruntjs.com

Grunt can’t do anything out of the box, because it’s a tool for running tasks that you should define. But for them to work it need to add configuration objects handled by plugins: from watching for changes to copying and concatenating files.

Its advantage is that there are thousands of plugins available, and you can always find one which does what you want. The big issue is that if you want to do something unusual, you can’t just write a plugin with plain JavaScript. Instead you’ll have to write your own plugin, which is not easy. 

$ npm install -g gulp-cli
$ gulp
gulpjs.com

Gulp is also a tool for defining and running tasks, but it defines tasks as JavaScript functions that return streams and promises, so you have a lot of flexibility in how you write your tasks.

Gulp also has a massive plugin library, but because its tasks are just plain functions, you’ll find you can also use a lot of vanilla node modules for those tasks too.

The biggest issue that Gulp has is that streams and promises can be hard to understand at first, but when you do, it will allow you to better apply custom tasks across your other code too.

Some thoughts

Currently there are many other build tools for JavaScript projects, but the ones we review here are the 4 most used. In fact, the option that I like most at the moment is Gulp, although the adoption of Webpack is growing a lot in the last time.

It all depends on the size of the project you have or the tool that the framework you are using already provides (or the system that the team which you are going to work with uses), we have different options to choose from.

My personal opinion is that Gulp is the best option if you have a project without a lot of code, and to choose Webpack if it's a monstrous project with a lot of code and files that must be managed.

Or even use both, because you can combine the module bundler tool with the task runner to automatize your development repetitive tasks!
